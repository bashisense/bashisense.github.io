注册登陆

获取设备地址

向同网段的6543端口发送广播报文，可以获取到设备的信息响应，在任意模式下，设备都会响应扫描请求，扫描请求/响应与设备的管理模式无关

{
    "action":"scan"
}

如果该地址段有WalOS在网设备（通常接到同一个交换机，在二层网络上相通，IP地址在同一个网段），会响应：

{
    "device-id":"device-id",                // 设备ID
    "hardware-version" : "v1.2.0",          // 硬件载体版本号
    "walos-version" : "v2.1.0",             // WalOS 操作系统版本号
    "protocol-version": "v2.2.0",           // 协议版本号
    "algm-vendor" : "bashi",                // 算法供应商
    "algm-version"  : "v1.1.0",             // 算法版本号
}

此时便可以获取到目标设备的ID、协议版本信息以及设备的IP地址信息；

独立模式登陆

设备配置成独立模式或另外几种模式时，设备均会启动WebAPI server，提供基于Http的API接口，此时对于接放的访问需要进行认证。

请求URL: http://dev-ip-addr:port/api/v3.x/auth 请求类型: POST 请求消息体:JSON 返回消息体:JSON, retcode = 0时正确， 非0时出错

接入认证
  • 请求：
{
    "action":"auth",

    "header":{
        "reqid":"192839",         // 透传值，设备内唯一
    },

    "body":{
        "user":"admin",
        "passwd":"admin-passwd"
    }
}
  • 响应：
{
    "retcode":0,

    "header":{
        "reqid":"192839",         // 透传值，设备内唯一
    },

    "body":{
        "token":"823lsefhw03slej",
        "timestamp":1560392020,
    }
}

token用于后续api请求时的http header的token字段，后续的访问必须设置 Token 字段

修改管理员密码
  • 请求：
{
    "action":"passwd",

    "header":{
        "reqid":"192839",         // 透传值，设备内唯一
    },

    "body":{
        "user":"admin",
        "passwd":"admin-new-passwd"
    }
}
  • 响应：
{
    "reqid":192839,         // 透传值，设备内唯一
    "retcode":0
}

终端模式登陆

终端模式设备分为以下几个状态： 1. 设备未连接状态：设备从未连接到管理平台，可能是网络配置或服务器地址配置问题导致设备未能接入管理平台，此时设备如果有显示屏，则显示网络配置信息，提示用户配置网络以使设备可以连接到管理平台； 2. 设备已连接状态：设备已接入到管理平台，网络处于连通状态，设备与管理平台进行以下操作：

  • 同步时间：设备向平台发送同步时间请求，请求格式见设备管理章节的设置时间；
  • 获取签名：如果设备没有签名密钥，则请求平台获取签名密钥并存储到本地；如果已有此签名密钥，则不执行此步骤；
  • 注册设备：如果设备没有激活码，则设备需要向平台注册，并获取激活码，存储到本地，同时显示激活码二维码，供App或小程序激活；如果已激活码则不执行此步骤；
  • 激活设备：设备轮询激活状态（每5秒一次），当获取设备密钥用于订阅；如果设备已经激活，则不执行此步骤；
  • 订阅消息：设备向平台发送订阅请求，订阅设备通知，设备进入正常工作状态；

当设备连接后，设备每5秒向平台发送一次心跳信息，连续5次平台没有响应心跳，则设备断开连接。

初次开机的设备会进行1/2/3/4/5步，已激活的设备直接进行第1/5步操作；

获取签名
  • 请求：
{
    "action":"get_sign_code",

    "header":{
        "reqid":"192839",         // 透传值，设备内唯一
    },

    "body":{
        "device-id":"device-id",    // 设备ID
        "timestamp":1560392020,     // 当时时间戳，计算签名时请转成字符串
        "nonce" : 19283892,         // 随机数，计算 签名时请转成字符串
        "signature":"xxxxx"         // 设备签名
    }
}

device-id 唯一标识一台设备 设备签名，sha1(sort(device-id、public-key、timestamp、nonce, “signature”))。sort的含义是将参数值按照字母字典排序，然后从小到大拼接成一个字符串。public-key是云平台的公钥，所有设备与第三方平台都有的。本接口计算签名时，需要把字符串常量 “signature”参与到计算之中，区别其他场景的签名。 public-key 采用base64编码后的串 如平台无需求确认设备是否为我司产品，可以忽略此命令的签名，直接返回sign_code用于后续签名即可，此sign_code由平台自己生成并管理

  • 响应：
{
    "retcode":0,                 // 获取成功

    "header":{
        "reqid":"192839",         // 透传值，设备内唯一
    },

    "body":{
        "sign_code":"xxxxxx",       // 签名码，字符串
    }
}

设备仅在第一次登陆平台时获取sign_code，并存储此值，直到设备被复位到出厂设备，后续一直使用此值签名；

注册设备
  • 请求：
{
    "action":"register",
    "header":{
        "reqid":"192839",         // 透传值，设备内唯一
    },

    "body":{
        "device-id":"device-id",    // 设备ID
        "timestamp":1560392020,     // 当时时间戳，计算签名时请转成字符串
        "nonce" : 19283892,         // 随机数，计算 签名时请转成字符串
        "signature":"xxxxx"         // 设备签名
    }
}

第三方平台，device-id 唯一标识一台设备 设备签名，sha1(sort(device-id、sign_code、timestamp、nonce, "register"))。sort的含义是将参数值按照字母字典排序，然后从小到大拼接成一个字符串。本接口计算签名时，需要把字符串常量 “register”参与到计算之中，区别其他场景的签名。 如平台无需求确认设备是否为已绑定到平台，可以忽略此命令的签名，直接返回active_code，此sactive_code由平台自己生成并管理

  • 响应：
{
    "retcode":0,                 // 获取成功
    "header":{
        "reqid":"192839",         // 透传值，设备内唯一
    },

    "body":{
        "active_code":"xxxxxx",     // 激活码，字符串
    }
}

设备会存储此激活码，每次注册到平台，返回的激活码应一致

激活设备
  • 请求：
{
    "action":"active",
    "header":{
        "reqid":"192839",         // 透传值，设备内唯一
    },

    "body":{
        "active_code":"xxxxxx",     // 激活码，字符串
    }
}
  • 响应：
{
    "retcode":0,                 // 激活成功
    "header":{
        "reqid":"192839",         // 透传值，设备内唯一
    },

    "body":{
        "secret":"xxxxxx",          // 凭证密钥，用于登陆系统
    }
}

设备存储secret值，用于登陆后的订阅命令； 已激活的设备，在连接入平台时，不再发送激活指令，直到设备解绑定； 激活返回失败时，设备应该轮询激活，时延设定为5秒直到返回激活成功，或者直接收到激活指令； 接收到激活指令后（body部分为secret），停止轮询，完成激活任务。应用平台retcode:0

订阅消息
  • 请求：
{
    "action":"subscribe",

    "header":{
        "reqid":"192839",         // 透传值，设备内唯一
    },

    "body":{
        "secret":"xxxxxx",      // 凭证密钥，用于登陆系统
    }
}
  • 响应：
{
    "retcode":0,                 // 激活成功
    "header":{
        "reqid":"192839",         // 透传值，设备内唯一
    },

    "body":{                        // 选填，订阅后平台可响应时间信息，设备自动校准时间
        "timestamp":15992901,       // 平台时间戳
        "timezone":-480             // 平台时区，分钟数偏移，东区为负西区为正，-480为东八区
    }
}

至此，设备进入被动模式，接收来自平台的指令，完成任务并返回响应 被动模式下，设备仅会向设备发送三种消息：1. 心跳，每5秒一次； 2. 同步时间，不定期，间隔1小时以上； 3. 事件上报； 除上述指令外，设备执行平台的指令

心跳消息
  • 请求：
{
    "action":"ping",

    "header":{
        "reqid":"192839",         // 透传值，设备内唯一
    },
}
  • 响应：
{
    "retcode":0,                 // 激活成功
    "header":{
        "reqid":"192839",         // 透传值，设备内唯一
    }
}
